# Тестовое задание
Небольшое API для мобильных приложений: регистрация/авторизация в системе.

Используемые библиотеки: Flask, SQL-Alchemy

Должно получиться два метода: POST /auth/signin/ и POST /auth/signup/.
POST /auth/signup/
Метод отвечает за регистрацию пользователя в системе. Принимает несколько параметров. 
Один из них type=[simple | facebook] — выбор способа регистрации, остальные зависят от выбранного способа (например,
для simple это email и password, для facebook это email, facebook_id, facebook_token). На выходе при отсутствии 
ошибок статус 200, при наличии — 400 и json с описанием ошибки. В базе создаётся новая запись о пользователе.
POST /auth/signin/
Метод отвечает за авторизацию пользователя. Принимает несколько параметров. Один из них type=[simple | facebook] — 
выбор способа авторизации, остальные зависят от выбранного способа (для simple это email и password, для facebook это facebook_token). 
На выходе при отсутствии ошибок статус 200 и возвращается токен авторизации, при наличии ошибки — 400 и json с описанием ошибки.

Что ещё должно быть:
возможность добавить ещё один способ авторизации (например, через instagram или twitter), либо отключить один из тех, что есть;
возможность заменить SQL-Alchemy (например на NoSQL-решение или peewee);
наличие тестов этих двух методов и идеально, если сперва будут написаны тесты, а потом уже реализация;

# Примечания реализации
При реализации я постарался уделить особое внимание масштабированию, через изоляцию слоя базы данных от логики авторизации. 
В итоге в качестве слоя БД можно использовать как SQL так и NoSQL решения, как документоориентированные, так и "ключ-значение".
В качестве примера документоориентированной базы реализован бэкенд на Redis через хеш массивы. Реализация для ключ-значения
аналогичная, только с ипользованием pickled.

Слой авторизации кофигурирует схему БД через список имен полей. SQL вариант на лету генерирует схему, а Redis использует ее
для операции hmget(нужен список полей).

При реализации использованы следующие допущения:
1. Все поля в БД строковые. Иначе нужно накручивать свой ORM сверху. Некоторые БД (например ElasticSearch) выполнят 
преобразование типов автоматически.
2. В задании не указано, как работает token. Я предположил, что это временный идентификатор сессии, и backend должен ументь
находить пользователя по этому идентификатору, а также иметь возможность его обновить не обновляя профиль пользователя.
